# API

Контракт серверного API. Транспортные решения и обоснования — в [architecture.md](./architecture.md).

Базовые принципы:

* Действия клиента — REST поверх HTTP. Поток событий сервер→клиент — одно WebSocket-соединение (`GET /ws`); по нему же идут эфемерные сигналы client→server (typing, read).
* Аутентификация — bearer-токен в заголовке `Authorization: Bearer <token>`.
* Тело сообщения для сервера непрозрачно: это `ciphertext`-blob, сервер не знает открытого текста (см. [encryption.md](./encryption.md)). В v1 шифрования ещё нет, но контракт уже исходит из того, что сервер текста не видит.
* Все идентификаторы — строки. Время — ISO-8601 UTC.
* Все REST-эндпоинты — под общим префиксом `/api/` (например `/api/auth/register`). WebSocket остаётся в корне (`/ws`). Префикс упрощает обратный прокси: одно правило `/api/` вместо правила на каждую группу.

> Статус: черновик v1. Поля помечены как обязательные минимально необходимые; список будет уточняться по мере реализации endpoint-ов.

## Аутентификация и устройства

Аутентификация v1 — упрощённая: регистрация по `username` + `password` **только при наличии валидного инвайт-кода** (открытой саморегистрации нет). Реальные ключи device/account встанут на это место позже без слома контракта (см. encryption.md). Пароль хранится только в виде хэша (argon2id). Инвайт-коды генерирует CLI-скрипт `npm run invite`.

### POST /api/auth/register
Создание учётной записи. Требует валидный неиспользованный инвайт-код.
* Тело: `{ "username": "...", "password": "...", "invite": "...", "deviceId": "<uuid>" }`
* `username` неизменяем после создания.
* Проверяет код (существует, не использован, не истёк); при успехе гасит код, создаёт аккаунт, выполняет авто-логин и регистрирует устройство.
* Ошибки: `400` — код невалиден/использован/истёк; `409` — `username` занят.
* Ответ: `{ "userId": "...", "username": "...", "accessToken": "..." }`

### POST /api/auth/login
* Тело: `{ "username": "...", "password": "...", "deviceId": "<uuid>" }`
* Токен — это сессия, привязанная к паре `(аккаунт, устройство)`.
* Незнакомый `deviceId` авто-регистрируется как новое устройство аккаунта; при этом эмитятся события `device.added` и `auth.attempt` (см. поток событий).
* Ответ: `{ "accessToken": "...", "userId": "..." }`

### POST /api/devices
Явная регистрация устройства текущего аккаунта (обычно не требуется — устройство регистрируется само при логине). Оставлено для будущей привязки ключа устройства.
* Тело: `{ "deviceId": "<uuid>", "devicePublicKey": "..." }`
* Ответ: `{ "deviceId": "..." }`

### GET /api/me
* Ответ: `{ "userId": "...", "username": "...", "devices": [...] }`

## Чаты

Объект чата (общий вид для всех ответов ниже):
```
{
  "chatId": "...",
  "type": "direct" | "group",
  "title": "...",            // для группы; у direct null
  "createdBy": "...",        // userId создателя чата; null у чатов до миграции 0004
  "participants": [ { "userId": "...", "username": "..." }, ... ],
  "lastMessage": {           // null, если сообщений ещё нет
    "messageId": "...",
    "senderId": "...",
    "ciphertext": "...",     // base64
    "ts": "2026-06-24T20:00:00Z"
  },
  "unreadCount": 3,
  "peerReadUpTo": "0",       // max last_read_message_id среди ДРУГИХ участников; сид статуса ✓✓
  "updatedAt": "2026-06-24T20:00:00Z"
}
```

### GET /api/chats
Список чатов аккаунта (отсортирован по `updatedAt` убыв.). Возвращает минимум, достаточный для рендера превью без открытия чата.
* Ответ: `{ "chats": [ <объект чата>, ... ] }`

### POST /api/chats
Создание чата.
* Тело (direct): `{ "type": "direct", "username": "..." }`
  * Дедупликация: если direct-чат с этим собеседником уже есть — возвращается он (`200`), иначе создаётся новый (`201`).
  * Ошибки: `404` — username не найден; `400` — direct с самим собой.
* Тело (group): `{ "type": "group", "title": "...", "members": ["username", ...] }`
  * Создатель добавляется в участники автоматически. `400`, если среди `members` есть несуществующий username.
* Ответ: объект чата. Всем участникам в outbox пишется событие `chat.created`.

### GET /api/chats/{chatId}
Доступно только участнику чата (иначе `404`).
* Ответ: объект чата.

### GET /api/chats/{chatId}/members
Список участников с признаком онлайн (снимок на момент запроса) и указанием создателя. Доступно только участнику (иначе `404`).
* Ответ: `{ "createdBy": "<userId|null>", "members": [ { "userId", "username", "online": true|false }, ... ] }`
* `online` — бинарный статус (есть ли у аккаунта живое WS-соединение). Живые изменения приходят событием `presence` (см. поток событий).

### POST /api/chats/{chatId}/members
Добавление участника в группу по username. Право — только у создателя чата (`created_by`).
* Тело: `{ "username": "..." }`.
* Ошибки: `400` — отсутствует username либо чат не группа; `403` — вызывающий не создатель; `404` — чат/пользователь не найден; `409` — пользователь уже участник.
* При успехе участник добавляется; новому участнику пишется `chat.created` (он подтягивает чат в список), уже состоящим — `chat.member_added` (обновляют состав/счётчик).
* Ответ: `201`, `{ "chatId": "...", "userId": "..." }`.

### DELETE /api/chats/{chatId}/members/{userId}
Удаление участника из группы. Право — только у создателя чата (`created_by`).
* Ошибки: `403` — вызывающий не создатель; `400` — чат не группа либо попытка удалить создателя; `404` — чат/участник не найден.
* При успехе участник удаляется; событие `chat.member_removed` пишется оставшимся участникам и самому удалённому (он убирает чат из своего списка).
* Ответ: `{ "chatId": "...", "userId": "..." }`

Все эндпоинты ниже доступны только участнику чата (иначе `404`). `ciphertext` — base64.

### GET /api/chats/{chatId}/messages
Ленивая подгрузка истории, обратный хронологический порядок (от новых к старым), пагинация по курсору. Курсор и порядок — по монотонному `messageId`.
* Query: `before={messageId}` (необязательно, для следующей страницы вглубь истории), `limit=N` (по умолчанию 50, максимум 100).
* Ответ:
```
{
  "messages": [ { "messageId", "senderId", "ciphertext", "blobIds", "ts", "editedAt", "deleted" }, ... ],
  "hasMore": true,
  "nextBefore": "<messageId>"
}
```
* У удалённого сообщения `deleted: true`, `ciphertext` пустой.
* `blobIds` — массив id вложений (см. «Блобы»); пустой, если вложений нет.

### POST /api/chats/{chatId}/messages
Отправка сообщения. Всем участникам в outbox пишется `message.new`, у чата обновляется `updatedAt`.
* Тело: `{ "clientMessageId": "...", "ciphertext": "...", "blobIds": ["...", ...] }`
* `clientMessageId` обеспечивает идемпотентность и оптимистичный UI: первая отправка — `201`; повтор с тем же id не создаёт дубликат и не шлёт событие повторно — `200` с тем же `messageId`.
* `blobIds` — необязательный список вложений (см. «Блобы»). Передаётся открыто: из зашифрованного `ciphertext` сервер ссылки не прочтёт, а они нужны ему для авторизации скачивания и подсчёта ссылок. Ключи расшифровки остаются внутри `ciphertext`. Каждый id должен существовать (иначе `400`); максимум 16 на сообщение.
* Ответ: `{ "messageId": "...", "clientMessageId": "...", "blobIds": [...], "ts": "..." }`

### PATCH /api/messages/{messageId}
Редактирование. Доступно только автору (`403` иначе, `404` если нет, `400` если уже удалено). Эмитит `message.edited`.
* Тело: `{ "ciphertext": "..." }`
* Ответ: `{ "messageId": "...", "editedAt": "..." }`

### DELETE /api/messages/{messageId}
Удаление. Доступно только автору (`403` иначе). Содержимое стирается, ставится `deleted`; идемпотентно. Эмитит `message.deleted`.
* Ответ: `{ "messageId": "..." }`

### POST /api/chats/{chatId}/read
Отметка о прочтении до указанного сообщения включительно. Маркер двигается только вперёд. Эмитит `message.read`.
* Тело: `{ "upToMessageId": "..." }`
* Ответ: `{ "ok": true }`

## Присутствие (онлайн)

Онлайн-статус — бинарный (есть ли живое WS-соединение у аккаунта), in-process на одной реплике сервера, без «был в сети N назад».

### GET /api/presence
Снимок онлайна со-участников вызывающего (тех, с кем он делит хотя бы один чат) — для сидирования клиента после коннекта. Дальше актуальность держится событиями `presence`.
* Ответ: `{ "online": [ "<userId>", ... ] }`

## Поток событий (WebSocket)

### GET /ws (WebSocket upgrade)
Единый канал событий аккаунта. Одно WS-соединение на устройство. Аутентификация — bearer-токен (заголовок при upgrade или первым сообщением hello).

`event` — общий класс, а не только сообщения. Сообщения чата — лишь одна из категорий; в тот же канал идут события уровня аккаунта/безопасности (вход с другого устройства, добавление устройства). Поэтому `chatId` присутствует только у chat-scoped событий, а `seq` — сквозной на уровне аккаунта.

Resume после (пере)коннекта:

* Первым сообщением клиент шлёт `{ "type": "hello", "lastSeq": <seq|0> }`.
* Сервер реплеит `events` с `seq > lastSeq` для этого аккаунта, затем шлёт управляющий маркер `{ "type": "synced" }` (без `seq` — это сигнал, а не событие outbox) и переходит к стримингу новых. Тот же механизм — для досинхронизации после пуша/офлайна (после wake-up клиент переоткрывает WS и шлёт hello с последним известным `seq`).
* Маркер `synced` отделяет историческую выгрузку («уже было») от живых событий («происходит сейчас»). Клиент применяет реплей к состоянию молча; уведомления безопасности и накрутку счётчиков делает только для событий после `synced`. Состояние (чаты/сообщения/непрочитанные) клиент авторитетно берёт из REST, а не из реплея.

Сообщения client→server (эфемерные, по тому же сокету):

* `{ "type": "typing", "chatId": "..." }`
* `{ "type": "read", "chatId": "...", "upToMessageId": "..." }`

Конверт события сервер→клиент (JSON-сообщение в сокете; для крупного бинаря допустимы бинарные фреймы):

```
{
  "type": "message.new",
  "seq": 12345,
  "chatId": "...",          // только у chat-scoped событий, иначе отсутствует
  "ts": "2026-06-24T20:00:00Z",
  "payload": { ... }        // зависит от type
}
```

Chat-scoped события (несут `chatId`):

* `message.new` — `payload: { messageId, senderId, clientMessageId, ciphertext, blobIds, ts }`
* `message.edited` — `payload: { messageId, ciphertext, editedAt }`
* `message.deleted` — `payload: { messageId }`
* `message.read` — `payload: { userId, upToMessageId }`
* `typing` — `payload: { userId }` (эфемерное: доставляется только подключённым сейчас, без записи в outbox и без `seq`)
* `chat.created` — `payload: { ... }`
* `chat.member_added` — участник добавлен в чат создателем — `payload: { chatId, userId }`. Идёт уже состоящим участникам (обновить состав/счётчик); новому участнику вместо этого приходит `chat.created`.
* `chat.member_removed` — участник удалён из чата создателем — `payload: { chatId, userId }`. Идёт оставшимся участникам (обновить состав) и самому удалённому (убрать чат из списка).
* `presence` — смена онлайн-статуса со-участника — `payload: { userId, online }` (эфемерное: только подключённым сейчас, без outbox и `seq`)

Account/security-scoped события (без `chatId`):

* `auth.attempt` — попытка входа с другого устройства — `payload: { deviceId, ip, userAgent, ts }`
* `device.added` — к аккаунту добавлено устройство — `payload: { deviceId, ts }`
* `device.removed` — `payload: { deviceId }`

## Блобы (вложения)

Крупные вложения (фото/видео/файлы), которые нельзя гнать inline в теле сообщения. Сервер хранит **непрозрачные блобы**, адресуемые по content-hash (sha256 содержимого — он же `blobId`); содержимого и ключей расшифровки не знает (в перспективе блоб шифруется клиентом, как и сообщения). Хранилище — за абстракцией `BlobStore`: локальная ФС в dev, объектный стор (MinIO) в prod/deploy; выбор драйвера — env `BLOB_STORE`.

Жизненный цикл: загрузить блоб → получить `blobId` → отправить сообщение с этим id в `blobIds`. До привязки к сообщению блоб доступен только загрузчику.

### POST /api/blobs
Загрузка блоба. Тело — сырые байты, `Content-Type: application/octet-stream` (поток, без JSON/base64). Сервер считает sha256 на лету, режет превышение `MAX_BLOB_SIZE` (`413`), дедуплицирует по хэшу (один и тот же контент не дублируется в сторе) и фиксирует загрузчика.
* Ответ: `201 { "blobId": "<sha256 hex>", "size": <bytes> }`
* Пустое тело — `400`.

### GET /api/blobs/{id}
Скачивание блоба потоком (`application/octet-stream`). Доступ: загрузчик блоба **или** участник чата, где есть неудалённое сообщение, ссылающееся на этот блоб (иначе `403`; `404`, если блоба нет).

## Превью ссылок (#32)

Браузер не может прочитать OG-теги со сторонней страницы (CORS), а сервер не читает ciphertext (не развернёт ссылку сам). Поэтому клиент отправителя просит сервер развернуть URL, получает метаданные + байты картинки и **вшивает превью в сообщение** вложением `kind:'link'` (в ciphertext). Получатель только рендерит карточку — без повторного фетча и без утечки своего IP стороннему сайту. Сервер при этом видит разворачиваемые URL'ы (неустранимая цена превью при E2EE).

### POST /api/unfurl
Развернуть ссылку. Сервер сам тянет страницу под **SSRF-защитой** (только http/https; отказ на приватные/loopback/link-local адреса с перепроверкой на каждом редиректе; лимит редиректов; таймаут; потолок размера; только `text/html`), парсит OpenGraph/`<title>`/`description`, скачивает og:картинку (тот же guard + лимит) и отдаёт её байтами base64 — клиент сам ужимает её в thumbnail (canvas), серверу image-библиотеки не нужны.
* Тело: `{ "url": "https://..." }`
* Ответ: `200 { "preview": { "url", "title", "description"?, "siteName"?, "image"?: { "mime", "dataBase64" } } }` либо `{ "preview": null }` — страница недоступна, заблокирована SSRF-защитой, не HTML или без заголовка.
* `400` — отсутствует `url` или схема не http/https.
* Env `UNFURL_ALLOW_PRIVATE=1` разрешает фетч приватных адресов (только dev/тесты).

## Push-подписки

Пуш — ортогональный транспорту канал «разбудить клиента», без содержимого сообщения. Получив wake-up, клиент переоткрывает WS и досинхронизируется через hello/`lastSeq`. Сервер шлёт wake-up получателю события, только если у того нет живого WS-соединения. В v1 реальной отправки в FCM/UnifiedPush ещё нет — серверная часть инкапсулирована как заглушка (точка интеграции с провайдерами).

### POST /api/push/subscriptions
Регистрация канала пуша устройства (FCM-токен или UnifiedPush-endpoint).
* Тело: `{ "deviceId": "...", "provider": "fcm" | "unifiedpush", "endpoint": "..." }`
* Подписать можно только собственное устройство (иначе `404`). Идемпотентно по паре `(deviceId, endpoint)`: повторная регистрация того же токена возвращает ту же подписку.
* Ошибки: `400` — отсутствует поле или неизвестный `provider`; `404` — устройство не принадлежит аккаунту.
* Ответ: `201` `{ "subscriptionId": "..." }`

### DELETE /api/push/subscriptions/{subscriptionId}
Удаляет только собственную подписку (привязка через устройство аккаунта). Идемпотентно.
* Ответ: `{ "ok": true }`
