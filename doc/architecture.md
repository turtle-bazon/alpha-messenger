# Архитектура и ключевые решения

Документ фиксирует принятые архитектурные решения серверной части и транспорта. Подробности контракта API — в [api.md](./api.md), модель данных — в [model.md](./model.md), шифрование — в [encryption.md](./encryption.md).

## Транспорт: SSE + REST

Для обмена выбрана связка:

* **REST** (поверх HTTP) — для всех действий клиента: отправка, редактирование, удаление сообщения, получение списка чатов и истории, отметка о прочтении.
* **SSE** (Server-Sent Events) — единый поток событий сервер→клиент.

`event` — общий класс, а не только «новое сообщение». Сообщение чата (`message.*`) — лишь одна из категорий нагрузки. Помимо них в тот же поток идут события уровня аккаунта/безопасности, не привязанные к чату: вход/попытка входа с другого устройства, добавление устройства и т.п. Поэтому `chatId` в конверте события — опциональное поле (есть только у chat-scoped событий), а лог событий и его `seq` — на уровне аккаунта, чтобы один `Last-Event-ID` покрывал всё разом.

Почему SSE, а не WebSockets:

* Логика мессенджера асимметрична. Действия клиента — это дискретные запрос-ответ, для них естественен REST. Persistent-соединение нужно только для пуша сервер→клиент — ровно то, что делает SSE.
* SSE даёт авто-reconnect и `Last-Event-ID` из коробки, работает поверх обычного HTTP/2, не требует upgrade-хендшейка и собственного фрейминга/heartbeat.
* Двунаправленность WebSockets нам не нужна; она потребовала бы ручной реализации reconnect и replay пропущенных событий.

## Один общий поток событий, а не по соединению на чат

На устройство открывается **один** SSE-поток (`GET /events`). Все события аккаунта идут в этот поток, каждое событие несёт `chatId`. Отдельные соединения по чатам не открываются.

Причины: лимит браузера ~6 соединений на домен (HTTP/1.1), экономия ресурсов сервера, единая логика reconnect/replay. По HTTP/2 SSE-поток и REST-запросы мультиплексируются в одно TCP-соединение, поэтому сервер должен отдаваться по HTTP/2.

## Push на мобильных устройствах ортогонален транспорту

Ключевой факт: ни SSE, ни WebSockets не выживают в фоне на Android (doze, ограничения батареи). Оба работают только пока приложение активно (foreground).

Доставка в фоне идёт через **отдельный канал пуша**:

* **FCM** — на устройствах с сервисами Google.
* **UnifiedPush** (unifiedpush.org) — на de-Googled устройствах.

Пуш используется как wake-up: он будит приложение, дальше оно досинхронизируется обычным REST-ом (`GET /events?since=...`). Сам пуш не переносит содержимое сообщения (сервер не знает текста, см. encryption.md) — он лишь сигнал «есть новое, синхронизируйся».

Следствие: выбор SSE vs WebSockets не зависит от UnifiedPush — это разные каналы. Поэтому UnifiedPush не является аргументом в пользу WebSockets.

Итоговая схема доставки:

* приложение активно → SSE;
* приложение в фоне → push (FCM/UnifiedPush) как wake-up → REST-синк.

## Хранилище: PostgreSQL

Для первой версии — PostgreSQL. Он закрывает:

* реляционную часть: аккаунты, устройства, чаты, участники, push-подписки;
* сообщения: строка с непрозрачным `ciphertext`-blob и метаданными (`chatId`, `senderId`, `ts`, `messageId`, `seq`);
* лог событий (outbox-таблица) — источник для SSE fan-out и для replay при reconnect.

Сервер никогда не видит открытый текст: тело сообщения для него — непрозрачный blob.

Non-goals v1: шардирование, переход на Cassandra/ScyllaDB, отдельное хранилище медиа. Это вопросы масштаба, к ним вернёмся при необходимости.

## Хранение сообщений на клиенте: в v1 не храним

В первой версии клиент не держит постоянный кэш сообщений. Открыли чат → подтянули историю с сервера → держим в памяти → закрыли чат → отпустили. Повторная загрузка (и повторная расшифровка в будущих версиях) при каждом открытии для v1 приемлема.

Список чатов при старте всё равно тянется с сервера; в нём приходит `lastMessage` каждого чата, чтобы рисовать превью без открытия чата.

Постоянный локальный кэш (IndexedDB, в т.ч. зашифрованный) — задача будущих версий.

## Технологический стек сервера

* Язык — TypeScript.
* HTTP-фреймворк — **Fastify**: нативная поддержка стримов (нужно для SSE), быстрый, хорошо ложится на TS.
* Доступ к БД — **node-postgres (pg)** с чистым SQL и миграциями, без тяжёлой ORM: схема простая, контроль над запросами полнее.

## Окружения и регистрация

Единица деплоя одна — `run/docker-compose.yaml` (сервис + PostgreSQL с persistent-волюмом). Тот же compose запускается локально при разработке и без изменений поднимается на VPS, когда дело дойдёт до совместного тестирования.

Регистрация — открытая саморегистрация (`username` + `password`), без подтверждений и без registration secret. Этого достаточно, чтобы после деплоя на VPS участники просто зарегистрировались сами. Контракт аутентификации спроектирован так, чтобы позже заменить упрощённую схему на ключи device/account без слома API.

## Идентификация и порядок сообщений

* У каждого сообщения два идентификатора: `clientMessageId` (генерит клиент, для идемпотентности отправки, дедупликации и оптимистичного UI) и `messageId` (присваивает сервер).
* Серверный лог событий монотонно нумеруется (`seq`). `seq` используется и как `Last-Event-ID` для SSE-resume, и для дельта-синка через REST после реконнекта/пуша.
