# Архитектура и ключевые решения

Документ фиксирует принятые архитектурные решения серверной части и транспорта. Подробности контракта API — в [api.md](./api.md), модель данных — в [model.md](./model.md), шифрование — в [encryption.md](./encryption.md).

## Транспорт: WebSocket + REST

Для обмена выбрана связка:

* **REST** (поверх HTTP) — для всех действий клиента: отправка, редактирование, удаление сообщения, получение списка чатов и истории, отметка о прочтении.
* **WebSocket** — единый канал приёма событий сервер→клиент плюс эфемерные сигналы client→server (typing, read).

Это индустриальный паттерн крупных мессенджеров: даже Discord и Slack отправляют сообщения через REST (HTTP POST), а WebSocket используют как канал приёма событий. У нас так же — действия по REST, поток событий по WS.

`event` — общий класс, а не только «новое сообщение». Сообщение чата (`message.*`) — лишь одна из категорий нагрузки. Помимо них в тот же поток идут события уровня аккаунта/безопасности, не привязанные к чату: вход/попытка входа с другого устройства, добавление устройства и т.п. Поэтому `chatId` в конверте события — опциональное поле (есть только у chat-scoped событий), а лог событий и его `seq` — на уровне аккаунта, чтобы один курсор покрывал всё разом.

Почему WebSocket, а не SSE:

* Двунаправленность: эфемерные высокочастотные сигналы (typing, read) идут client→server прямо по сокету, а не отдельными HTTP POST на каждый.
* Нативные бинарные фреймы: ciphertext-блобы можно передавать без base64.
* Меньше накладных расходов на сообщение после установления соединения; ниже задержка.

Цена по сравнению с SSE: авто-reconnect и resume SSE давал из коробки — на WS реализуем сами (см. ниже hello/resume). Работа небольшая: модель `seq`/outbox у нас уже есть, на ней resume и строится.

## Один общий канал событий, а не по соединению на чат

На устройство открывается **одно** WS-соединение. Все события аккаунта идут в него, каждое chat-scoped событие несёт `chatId`. Отдельные соединения по чатам не открываются.

Причины: экономия ресурсов сервера и единая логика reconnect/replay.

Resume после (пере)коннекта: клиент при подключении шлёт hello с `lastSeq` (последний полученный `seq`), сервер реплеит `events WHERE seq > lastSeq` для этого аккаунта, затем переходит к стримингу новых. Тот же механизм используется для досинхронизации после пуша/офлайна.

## Push на мобильных устройствах ортогонален транспорту

Ключевой факт: ни SSE, ни WebSockets не выживают в фоне на Android (doze, ограничения батареи). Оба работают только пока приложение активно (foreground).

Доставка в фоне идёт через **отдельный канал пуша**:

* **FCM** — на устройствах с сервисами Google.
* **UnifiedPush** (unifiedpush.org) — на de-Googled устройствах.

Пуш используется как wake-up: он будит приложение, дальше оно переоткрывает WS и досинхронизируется через hello с `lastSeq`. Сам пуш не переносит содержимое сообщения (сервер не знает текста, см. encryption.md) — он лишь сигнал «есть новое, синхронизируйся».

Следствие: выбор транспорта (WS или SSE) не зависит от UnifiedPush — это разные каналы. Поэтому UnifiedPush не является аргументом ни за, ни против WebSockets.

Итоговая схема доставки:

* приложение активно → SSE;
* приложение в фоне → push (FCM/UnifiedPush) как wake-up → REST-синк.

## Хранилище: PostgreSQL

Для первой версии — PostgreSQL. Он закрывает:

* реляционную часть: аккаунты, устройства, чаты, участники, push-подписки;
* сообщения: строка с непрозрачным `ciphertext`-blob и метаданными (`chatId`, `senderId`, `ts`, `messageId`, `seq`);
* лог событий (outbox-таблица) — источник для SSE fan-out и для replay при reconnect.

Сервер никогда не видит открытый текст: тело сообщения для него — непрозрачный blob.

Non-goals v1: шардирование, переход на Cassandra/ScyllaDB, отдельное хранилище медиа. Это вопросы масштаба, к ним вернёмся при необходимости.

## Хранение сообщений на клиенте: в v1 не храним

В первой версии клиент не держит постоянный кэш сообщений. Открыли чат → подтянули историю с сервера → держим в памяти → закрыли чат → отпустили. Повторная загрузка (и повторная расшифровка в будущих версиях) при каждом открытии для v1 приемлема.

Список чатов при старте всё равно тянется с сервера; в нём приходит `lastMessage` каждого чата, чтобы рисовать превью без открытия чата.

Постоянный локальный кэш (IndexedDB, в т.ч. зашифрованный) — задача будущих версий.

## Технологический стек сервера

* Язык — TypeScript.
* HTTP-фреймворк — **Fastify**; WebSocket — через `@fastify/websocket`. Быстрый, хорошо ложится на TS.
* Доступ к БД — **node-postgres (pg)** с чистым SQL и миграциями, без тяжёлой ORM: схема простая, контроль над запросами полнее.

## Окружения и регистрация

Единица деплоя одна — `run/docker-compose.yaml` (сервис + PostgreSQL с persistent-волюмом). Тот же compose запускается локально при разработке и без изменений поднимается на VPS, когда дело дойдёт до совместного тестирования.

Регистрация — **только по инвайт-коду** (`username` + `password` + `invite`), открытой саморегистрации нет. Коды одноразовые, гасятся при использовании.

Генерация кодов в v1 — CLI-скрипт `npm run invite` (печатает ссылку `/<client>/register?invite=CODE`). Системы ролей/прав на этом этапе нет: кто имеет доступ к серверу, тот и создаёт инвайты. Таблица `invites` спроектирована с расчётом на будущее «инвайт от пользователя» (колонка `created_by`) — это добавляется одним эндпоинтом поверх той же таблицы, без введения ролей. Контракт аутентификации спроектирован так, чтобы позже заменить упрощённую схему на ключи device/account без слома API.

## Идентификация и порядок сообщений

В системе две монотонные сквозные величины, не путать:

* `messageId` — идентичность и порядок сообщений. Глобальная `IDENTITY`-последовательность; служит и идентификатором сообщения, и курсором порядка/пагинации истории чата (`before=messageId`). У сообщения есть ещё `clientMessageId` — его генерит клиент для идемпотентности отправки, дедупликации и оптимистичного UI.
* `seq` — курсор потока событий. Глобальная `IDENTITY`-последовательность в логе событий (outbox). Клиент передаёт последний полученный `seq` в hello при (пере)коннекте WS, сервер реплеит всё с бо́льшим `seq`. Тот же `seq` используется для досинхронизации после пуша/офлайна.

Обе величины глобальные (а не счётчики на аккаунт): они монотонны, для каждого аккаунта дают возрастающую подпоследовательность (с «дырами» от чужих записей, что для курсоров неважно), и не создают contention на per-account счётчике.
