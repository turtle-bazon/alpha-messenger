# #70 — Кеш сообщений

## Проблема
При открытии чата или возврате к нему сообщения загружаются с сервера каждый раз. Это создаёт задержку и лишние запросы, хотя порядок сообщений гарантированно последовательный (sequence).

## Решение
Реализовать локальный кеш сообщений в `IndexedDB`, который:

### Хранение
- Object store: `messages`, индекс по `chatId`
- Формат: отсортированный по `seq` массив объектов `Message`
- Хранить последние N сообщений (например, 100) для каждого чата

### Заполнение кеша
- При загрузке чата с сервера — сохранять сообщения в кеш
- При скролле вверх (загрузка истории) — дописывать старые сообщения в кеш
- При получении WebSocket-событий (`message.new`, `message.edited`, `message.deleted`) — обновлять кеш

### Чтение из кеша
- При открытии чата — показывать сообщения из кеша мгновенно
- Параллельно запрашивать свежие данные с сервера (optimistic update)
- Если кеш пуст — загружать с сервера как обычно

### Инвалидация
- При дисконнекте WebSocket или ошибке соединения — помечать кеш как potentially stale
- При логине/регистрации — очищать кеш
- При ручном обновлении (pull-to-refresh) — полная перезагрузка

### Границы
- Кеш — производительная оптимизация, а не офлайн-режим
- Данные могут быть неактуальны на время жизни кеша
- Старые сообщения (при скролле) дописываются в кеш, но не дублируются

## Интерфейс
```typescript
interface MessageCache {
  get(chatId: string): Message[];
  set(chatId: string, messages: Message[]): void;
  append(chatId: string, messages: Message[]): void; // для истории
  updateMessage(chatId: string, message: Message): void;
  deleteMessage(chatId: string, messageId: string): void;
  clear(chatId?: string): void;
}
```
