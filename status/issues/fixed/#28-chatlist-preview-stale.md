# #28 — Баг: превью последнего сообщения в ChatList не обновляется

[Баг] Превью последнего сообщения в ChatList не обновляется при получении новых сообщений через WebSocket. Наблюдение: в ChatList у чата отображается устаревшее превью с old временем, хотя в самом чате уже видны более новые сообщения. Это значит, что поле lastMessage в объекте чата не обновляется при инкрементальном обновлении через WS.

Клоду: проверить весь пайплайн — как WS-событие message приходит на клиент, как обновляется состояние чатов (chats state), как ChatList читает lastMessage из этого состояния, и где именно происходит (или не происходит) обновление lastMessage при новом сообщении. Возможно lastMessage обновляется только при полной перезагрузке списка чатов (GET /chats), но не при инкрементальном WS-обновлении.

## Решение

Основная гипотеза («lastMessage обновляется только при полной перезагрузке, а инкрементально через WS — нет») **не подтвердилась**. Обработчик `message.new` в `HomeScreen` обновляет `lastMessage` объекта чата по живому событию начиная с самого раннего рабочего коммита клиента (задолго до заведения этой задачи). Прогон функциональных сценариев подтвердил, что превью обновляется во всех типичных случаях:

* приём сообщения при закрытом и при открытом чате;
* превью у самого отправителя (через WS-эхо);
* оффлайн-сообщения + повторный вход (свежее из GET /chats);
* reconnect без перезагрузки (потеря/восстановление сети) — превью догоняет реплей;
* приём изображения → «📷» (уже покрыто `image.spec.ts`).

То есть как описано — баг невоспроизводим.

**Что нашли и поправили.** Единственная реальная гонка в пайплайне — ветка `idx < 0` в обработчике `message.new` (`HomeScreen.tsx`): когда чата ещё нет в списке и в него сразу прилетает несколько сообщений, на каждый `message.new` запускается отдельный `getChat()`. «Победителем» становился тот, что разрешился первым, и он мог нести устаревший снимок `lastMessage` — превью застревало на раннем сообщении, хотя в открытом чате (через `getMessages`) видны все. Это совпадает с формулировкой «в чате видны более новые сообщения, а превью старое».

Исправление: при добавлении/слиянии чата из `getChat` превью берётся как самое свежее (по `message_id`) из загруженного снимка, уже лежащего в списке элемента и самого события (`newestPreview`). Так превью не зависит от того, какой из параллельных `getChat` разрешился первым.

Регрессия: `e2e/preview-stale.spec.ts` — инкрементальное обновление превью у обоих участников и всплеск сообщений во вновь появившийся чат (превью = последнее сообщение серии).

Итог: проблема в описанной формулировке не воспроизводится; поправили найденную гонку getChat и закрыли регрессом.
