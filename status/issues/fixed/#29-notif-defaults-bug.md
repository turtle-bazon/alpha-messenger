# #29 — Некорректные дефолты уведомлений после очистки localStorage

[Баг] После удаления всех ключей localStorage и повторного входа — настройки уведомлений ведут себя странно. На скрине видно: «Звук» включён, «Уведомления браузера» выключены с подсказкой «Уведомления заблокированы в настройках браузера», но ключей `alpha.notif.sound` и `alpha.notif.browser` в localStorage вообще нет (только auth-ключи).

Проблемы:
1. Дефолтное поведение при отсутствии ключей: `readFlag()` считает отсутствие = `true` (включено). Но в UI показывается что браузерные выключены. Непонятно почему — возможно `ensureBrowserPermission()` при входе определяет `denied` и молча переключает, но не пишет в localStorage.
2. Если браузер заблокировал уведомления (`Notification.permission === 'denied'`), `ensureBrowserPermission()` не сохраняет это в localStorage. Каждый новый логин = повторная попытка запроса (бессмысленная).
3. Ключи `alpha.notif.sound` / `alpha.notif.browser` не появляются в localStorage при первом входе (дефолт не сидит явно).

Клоду: проверить весь пайплайн дефолтов уведомлений. При входе: если ключей нет — явно записать дефолты (`'1'` для обоих). Если `Notification.permission === 'denied'` — записать `'0'` для browser и больше не спрашивать. Убедиться что UI корректно отражает состояние.

## Решено

Добавлена явная инициализация дефолтов `initNotifDefaults()` в `util/notifications.ts`, вызывается из `HomeScreen` при входе перед `ensureBrowserPermission()`:
* `sound`: если ключа нет — сидим `'1'`.
* `browser`: если `getPermission() === 'denied'` — принудительно `'0'` (фиксируем выключенным, не включаем и не спрашиваем впредь); иначе, если ключа нет — `'1'`.

Это убирает корневой рассинхрон из скрина: при заблокированном разрешении в хранилище теперь лежит `browser='0'` (а не «логически вкл, но в UI выкл»), и ключи перестают быть пустыми после очистки localStorage. Пользовательский явный выбор (`'0'`/`'1'`) не перетирается, кроме принудительного `'0'` при `denied`.

По пунктам задачи:
1. Рассинхрон устранён: состояние хранилища и UI совпадают (при `denied` оба = выкл).
2. Повторного запроса при `denied` и так не было — `ensureBrowserPermission()` выходит на `permission !== 'default'`; теперь при `denied` ещё и не доходит до запроса из-за `!getNotifPrefs().browser`.
3. Ключи теперь явно сидятся при входе.

Сознательный компромисс (истина конкретна): если позже разблокировать уведомления в браузере, `browser` останется `'0'` до ручного включения в меню — ровно то, что просит формулировка «denied → больше не спрашивать».

Покрыто e2e `e2e/notif-defaults.spec.ts` (granted → оба `'1'`; denied → `browser='0'` + выключенный недоступный тумблер и подсказка). Регресс 31/31.
