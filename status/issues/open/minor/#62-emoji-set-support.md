# #62 — Поддержка разных наборов эмодзи

Пользователи должны иметь возможность выбирать набор эмодзи из предустановленных или создавать собственные.

## Текущее состояние
- Используется один встроенный набор эмодзи (системный)
- Нет возможности выбора или настройки набора

## Ожидаемое поведение

### Предустановленные наборы
- Набор по умолчанию (сейчас используемый)
- Telegram-style набор
- Apple-style набор
- Google-style набор
- Выбор набора в настройках

### Кастомные наборы (футуристика)
- Пользователь может создать свой набор эмодзи
- Источник: URL изображения или загрузка файла
- Каждый эмодзи в наборе: `name` (строка для поиска) + `image` (URL/данные)
- Наборы привязаны к аккаунту, хранятся на сервере

### Совместное использование наборов
- Пользователь может опубликовать свой набор (сделать публичным)
- Публичные наборы доступны другим пользователям для установки
- Каталог публичных наборов (как в Telegram stickers)
- Можно добавить чужой публичный набор в свой список

## Реализация (предварительная)

### Сервер
- Таблица `emoji_sets` (set_id, user_id, name, is_public, created_at)
- Таблица `emoji_set_items` (set_id, position, name, image_url/blob_id)
- Таблица `user_emoji_sets` (user_id, set_id, position) — порядок наборов у пользователя
- CRUD эндпоинты для наборов
- Эндпоинт каталога публичных наборов

### Клиент
- Экран выбора набора в настроеках
- Экран создания/редактирования набора
- Каталог публичных наборов с возможностью установки
- Обновление EmojiPicker для поддержки разных наборов
- Переключение набора в реальном времени

### Хранение
- Маленькие эмодзи (spans/inline) — base64 или URL
- Большие эмодзи (sticker-like) — blob storage (уже есть)

## Приоритет
normal

## Примечание
При реализации продумать архитектуру так, чтобы было просто добавлять новые типы наборов (animated, 3D, и т.п.) без переписывания базы.
