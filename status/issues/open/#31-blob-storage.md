# #31 — Хранилище больших вложений (медиа/файлы)

Поддержка крупных вложений (фото/видео/произвольные файлы), которые нельзя гнать
inline в теле сообщения (потолок ≈128 КБ, см. `doc/architecture.md`). Сервер
хранит **непрозрачные блобы**, адресуемые по content-hash; в перспективе они
шифруются клиентом так же, как любые сообщения (E2EE) — сервер содержимого и
ключей не знает.

## Технология хранения
Абстракция `BlobStore` (интерфейс) с двумя драйверами:
* `fs` — локальная content-addressed ФС (шардинг `ab/cd/<hash>`), дефолт в dev и
  тестах: новый сервис не нужен, только каталог/volume.
* `s3` — S3-совместимый объектный стор (**MinIO** в prod/deploy; путь в облако
  S3/R2/B2 потом — без правки кода). `@aws-sdk/client-s3`, path-style.

Выбор драйвера — env `BLOB_STORE=fs|s3`.

## Сделано (этап 1 — сервер)
* Миграция `0005_blobs.sql`: `blobs` (blob_id=sha256, size), `blob_owners`
  (загрузчики — у одного дедуплицированного блоба их может быть несколько),
  `message_blobs` (связь сообщение↔блоб).
* `POST /api/blobs` — стрим сырого тела (`application/octet-stream`), sha256 на
  лету (= blob_id), лимит `MAX_BLOB_SIZE`, дедуп, запись владельца.
* `GET /api/blobs/{id}` — отдаёт байты. Доступ: загрузчик блоба **или** участник
  чата, где есть неудалённое сообщение, ссылающееся на блоб.
* `POST /chats/{id}/messages` — опц. поле `blobIds[]` (открытые ссылки;
  серверу нужны явно, т.к. из ciphertext их не прочесть). Эхо `blobIds` в
  ответе, в `message.new` и в истории сообщений.
* `BlobStore` (`fs`/`s3`), инициализация на старте (каталог / бакет с ожиданием
  MinIO). MinIO добавлен в prod и deploy compose; dev — на `fs`.
* Тест `test/blobs.test.ts`: загрузка/дедуп, доступ по членству, 403 постороннему,
  404 на неизвестный, 400 на несуществующий blobId при отправке, 413 на лимит.

## Follow-up (отдельные этапы)
* GC/refcount: чистка блобов без ссылок (после жёсткого удаления сообщений) и из
  стора, и из БД. Сейчас soft-delete снимает доступ, но блоб остаётся.
* Защита от bandwidth-DoS: при превышении лимита тело сейчас дочитывается до
  конца (чтобы корректно закрыть запрос). Нужна отсечка на уровне прокси/
  соединения для гигантских загрузок.
* Клиентская часть: загрузка с прогрессом, шифрование блоба, inline-thumbnail +
  полноразмер по ссылке, рендер вложений.
* PATCH сообщения пока не меняет набор вложений (attachments задаются при отправке).
* presigned URL для прямой отдачи из S3 (разгрузить сервер) — на вырост.
