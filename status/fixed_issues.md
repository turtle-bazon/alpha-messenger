1. Все серверные REST-эндпоинты объединены под общим префиксом /api/ (app.ts: api-роуты регистрируются плагином с prefix:'/api', /ws остаётся в корне). Поправлены клиент (apiUrl добавляет /api — единый источник префикса), серверные тесты, e2e-хелпер и doc/api.md. Apache-прокси упрощён: одно правило ProxyPass /api/ вместо семи. Проверено: серверные тесты 8/8, e2e-регресс 9/9.
2. Клиент за обратным прокси больше не ходит в localhost:3000. В прод-сборке VITE_API_URL='' → URL относительные (запросы на тот же origin, что и страница; CORS не возникает, образ не привязан к домену). wsUrl() при пустом API_URL строится из window.location (схема ws/wss по протоколу страницы). Build-arg VITE_API_URL добавлен в client Dockerfile (дефолт localhost:3000 для локальной run/prod-сборки) и проброшен пустым в CI (.github/workflows/docker-image.yml) для публикуемого образа.
3. Статус прочтения ✓✓ деградировал в ✓ при повторном открытии чата (или переподключении WS). Причина: позиция прочтения собеседником (readUpTo) жила только в эфемерном состоянии клиента и сеялась лишь из live-события message.read — при перемонтировании Conversation сбрасывалась в 0, а старое событие повторно не реплеилось (resume отдаёт только события после lastSeq). Сервер хранит last_read_message_id, но не отдавал чужую позицию. Решение: loadChat отдаёт peerReadUpTo (max last_read_message_id среди других участников), Conversation сидит им readUpTo при открытии, HomeScreen держит peerReadUpTo в объекте чата актуальным по live message.read (даже у закрытых чатов). Покрыто e2e interactions.spec.ts («статус ✓✓ переживает переоткрытие чата»).
4. Время в мета-строке пузыря ломалось посимвольно («20:1 / 3»): у .bubble стоит overflow-wrap: anywhere, что снижает min-content потомков и позволяет браузеру переносить время. Решение: .bubble-meta { white-space: nowrap } — мета (время/статус) всегда в одну строку.
