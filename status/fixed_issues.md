Поправлены клиент (apiUrl добавляет /api — единый источник префикса), серверные тесты, e2e-хелпер и doc/api.md. Apache-прокси упрощён: одно правило ProxyPass /api/ вместо семи. Проверено: серверные тесты 8/8, e2e-регресс 9/9.
2. Клиент за обратным прокси больше не ходит в localhost:3000. В прод-сборке VITE_API_URL='' → URL относительные (запросы на тот же origin, что и страница; CORS не возникает, образ не привязан к домену). wsUrl() при пустом API_URL строится из window.location (схема ws/wss по протоколу страницы). Build-arg VITE_API_URL добавлен в client Dockerfile (дефолт localhost:3000 для локальной run/prod-сборки) и проброшен пустым в CI (.github/workflows/docker-image.yml) для публикуемого образа.
3. Статус прочтения ✓✓ деградировал в ✓ при повторном открытии чата (или переподключении WS). Причина: позиция прочтения собеседником (readUpTo) жила только в эфемерном состоянии клиента и сеялась лишь из live-события message.read — при перемонтировании Conversation сбрасывалась в 0, а старое событие повторно не реплеилось (resume отдаёт только события после lastSeq). Сервер хранит last_read_message_id, но не отдавал чужую позицию. Решение: loadChat отдаёт peerReadUpTo (max last_read_message_id среди других участников), Conversation сидит им readUpTo при открытии, HomeScreen держит peerReadUpTo в объекте чата актуальным по live message.read (даже у закрытых чатов). Покрыто e2e interactions.spec.ts («статус ✓✓ переживает переоткрытие чата»).
4. Время в мета-строке пузыря ломалось посимвольно («20:1 / 3»): у .bubble стоит overflow-wrap: anywhere, что снижает min-content потомков и позволяет браузеру переносить время. Решение: .bubble-meta { white-space: nowrap } — мета (время/статус) всегда в одну строку.
5. Двойной счёт непрочитанного на холодном старте/переподключении (бейдж показывал, напр., 3 вместо 1 после reload — флапало). Причина: в обработчике message.new флаг ws.isLive() вычислялся ВНУТРИ отложенного setChats-апдейтера; к моменту его выполнения React мог уже прогнать обработчик маркера 'synced' (live=true), и реплейные сообщения принимались за live, накручивая счётчик. Решение: фиксируем live = ws.isLive() в момент приёма события (синхронно в обработчике), а не в апдейтере. Дополнительно укреплён WsClient: при reconnect/StrictMode-двойном эффекте старый сокет отвязывается (detach: снятие onmessage/onclose + close) и его «поздние» сообщения игнорируются (guard this.ws !== ws), чтобы вытесненный сокет не портил общее состояние live/lastSeq. Покрыто e2e two-clients.spec.ts (переписка двух клиентов со свёртыванием/восстановлением окон; --repeat-each стабильно зелёный).
6. Reconnect/reload реплеил всю историю outbox с нуля и рисковал выдать её за live (уведомления/звуки/бейджи). Причина: клиент не сохранял курсор потока между сессиями — при создании WsClient lastSeq=0, и сервер реплеил всё. Решение: lastSeq сохраняется в localStorage per-account (session.ts: getLastSeq/setLastSeq, чистится в clearSession), WsClient зовёт onSeqAdvance при продвижении курсора, HomeScreen сидит WsClient сохранённым lastSeq и пишет его обратно — после reload реплеится только пропущенное. Граница replay/live (isLive по 'synced') уже отсекала историю от живых событий — это сохранено. Покрыто e2e reconnect.spec.ts (после reload security-уведомление повторно не всплывает, lastSeq в localStorage не сбрасывается).

10. Визуальная полировка UI до уровня современного мессенджера (ориентир — десктопный Telegram). Подход — меняли свои стили, без импорта чужого CSS; у Telegram взяли только эталонные значения (палитра, метрики). Вся палитра вынесена в CSS-переменные (единая точка темы). Сайдбар: тень+линия отделяют его от области чата, hover/active с переходом, поле поиска с иконкой-лупой и локальной фильтрацией списка по названию, аватары с тенью, badge непрочитанных. Переписка: мягкий градиент+точечный узор фона (чистый CSS), пузыри с радиусом/тенью/градиентом исходящих, хвостик у последнего сообщения группы (треугольник псевдоэлементом, без SVG), группировка подряд идущих сообщений одного автора, разделители дат («Сегодня»/«Вчера»/«24 июня»), синий ✓✓ (прочитано). Композер: округлённое поле со скрепкой, круглая кнопка-самолётик, появляющаяся анимацией только при наличии текста, анимированные точки «печатает». Решение по аватарам: оставлены круглые (как в Telegram Desktop), вопреки буквальной формулировке задачи про «скруглённый квадрат». 11. [Баг] Список чатов «мигал»/многократно перерисовывался при логине. Причины: (а) сервер шлёт каждое replay-событие отдельным WS-кадром → отдельный onmessage → отдельный setChats → отдельный ререндер (на холодном старте с большой историей — десятки перерисовок); (б) ws.connect() стартовал параллельно с getChats(), и replay-обработчики chat.created/message.new дёргали getChat/getPresence на каждый чат, хотя список уже авторитетен из getChats. Решение: (1) WsClient буферизует все события реплея и применяет их одним синхронным пакетом на маркер 'synced' (React 18 батчит в один ререндер; live включается после флаша, поэтому реплей по-прежнему не считается за live); (2) HomeScreen коннектит WS только после resolve getChats() — реплей ложится на готовый список; (3) точечные getChat/getPresence в обработчиках chat.created/member_added/member_removed вызываются только для реально отсутствующего чата либо только вживую (ws.isLive()), на реплее состав/presence уже из getChats + единого снимка по 'synced'. Покрыто новым e2e login-replay.spec.ts (после повторного входа с lastSeq=0 точечный getChat не вызывается ни разу, список восстановлен из getChats). Регресс e2e 20/20.
12. [Баг] Сообщения задваивались в открытой переписке — рандомно, после переключения чатов (дублировались и отправленные, и полученные, целым блоком). Причина: дубль с одинаковым messageId мог породить только один путь — ленивая подгрузка истории по скроллу вверх (onScroll), где страница сливалась в список через `merged.push(...)` БЕЗ дедупликации (все остальные пути идут через upsert, дедуп по messageId). Перекрытие диапазонов возникало двумя способами: (а) защита от повторного входа `loadingMore` жила в React-state — несколько scroll-событий в одном тике читали старое (false) значение, проскакивали guard и грузили одну и ту же страницу дважды; (б) Conversation монтировался без key={chatId} — при переключении чатов инстанс переиспользовался, и пока новый чат грузился, hasMore/nextBefore ещё держали значения прошлого чата, из-за чего onScroll по пустому списку (scrollTop≈0) подгружал текущий чат по чужому nextBefore. Решение: (1) key={chatId} на Conversation в HomeScreen — переключение чата даёт свежий маунт (hasMore=false, nextBefore=null), стейт прошлого чата не протекает; (2) guard повторного входа переведён на ref (loadingMoreRef) — синхронно отсекает параллельные scroll-события; (3) слияние страницы дедуплицирует по messageId (страховка от любого перекрытия диапазонов). Покрыто новым e2e message-dedup.spec.ts (60 сообщений, два scroll-события в одном тике → ровно 60 без дублей; падает на старом коде, проходит на новом). Регресс e2e 21/21.
13. Уведомления о непрочитанных сообщениях в web-клиенте (три механизма, по умолчанию всё включено): (а) браузерное системное уведомление при входящем; (б) счётчик непрочитанных в title вкладки («(3) alpha»); (в) короткий звук. Звук и попап срабатывают, только когда вкладка не активна (document.hidden или потерян фокус — как в Telegram Web); при активной вкладке хватает badge в списке и счётчика в title. Реализация: util/notifications.ts (настройки sound/browser в localStorage с дефолтом «включено», запрос Notification.permission, playSound через WebAudio без ассета, setUnreadBadge для title, notifyIncoming с проверкой видимости вкладки), компонент NotificationSettings (колокольчик в шапке + выпадающее меню с двумя тумблерами; включение браузерных запрашивает системное разрешение, при denied тумблер заблокирован с подсказкой), иконки IconBell/IconBellOff. Браузерные попапы работают «из коробки»: приложение само запрашивает Notification.permission при входе (ensureBrowserPermission в HomeScreen), если настройка включена, а разрешение ещё 'default' — иначе попапы молча не показывались бы (permission по умолчанию не granted), а тумблер прятал бы запрос. Тумблер браузерных честно отражает browser && permission==='granted'. В HomeScreen: вызов notifyIncoming в обработчике message.new (только живое чужое сообщение, имя чата из chatsRef) и эффект суммы unreadCount → setUnreadBadge. Отсекаются собственные и реплейные события (как у security-уведомлений). Покрыто новым e2e notifications.spec.ts (badge в title + перехват браузерного уведомления при неактивной вкладке через подмену Notification; переключение и сохранение настроек между сессиями). Регресс e2e 23/23.
